Network Security: Goals & Threat Model

Confidentiality · Authentication · Integrity · Availability · MitM · Replay · Eavesdropping · DoS

EXPLAIN

Network security is about ensuring that communication between parties achieves confidentiality, authentication, message integrity, and access/availability. Real security must be designed against a concrete threat model — who is the attacker, what can they observe or modify, what resources do they have?

  • Confidentiality: only sender and intended receiver should understand message content → requires encryption
  • Authentication: sender and receiver must confirm each other's identity — is Alice really talking to Bob?
  • Message integrity: content must not be altered in transit (accidentally or maliciously) → requires MACs or digital signatures
  • Access and availability: services must remain accessible to legitimate users → defend against DoS attacks
  • Non-repudiation: sender cannot deny having sent a message → requires digital signatures
  • Attacker capabilities (Dolev-Yao threat model): attacker (Trudy) can intercept, inject, replay, impersonate
  • Eavesdropping: passive listening on the channel; defeats confidentiality if traffic is unencrypted
  • Man-in-the-Middle (MitM): attacker intercepts and possibly modifies traffic between two parties
  • Replay attack: attacker records a valid message and re-sends it later; encryption alone does NOT prevent this
  • Denial of Service (DoS): flood a host or network with traffic to exhaust resources
INTERACTIVE SIMULATION
THINK OF IT LIKE…

Think of sending a letter. Confidentiality is putting it in a locked box. Authentication is the wax seal proving who sent it. Integrity is tamper-evident tape showing if anyone opened it. Availability is making sure the mail truck isn't blocked. Trudy is a spy who can copy, forge, delay, or replace letters — your security mechanisms need to defeat all of these.

Symmetric Key Cryptography (DES, AES)

Shared secret · Block cipher · DES · AES · CBC mode · CTR mode · Key distribution problem

Complete Section 1 to unlock

EXPLAIN

Symmetric key cryptography uses the same secret key for both encryption and decryption. Both parties must share a secret key before communicating. Modern symmetric ciphers are efficient and can encrypt gigabits per second in hardware. The core challenge is key distribution — how do two parties securely agree on a shared key without already having one?

  • Symmetric key: K_A = K_B = K_AB — same key encrypts and decrypts
  • Block cipher: encrypts fixed-size blocks. DES: 64-bit blocks, 56-bit key; AES: 128-bit blocks, 128/192/256-bit key
  • DES (Data Encryption Standard): obsolete — 56-bit key is exhaustively breakable in hours with modern hardware
  • 3DES: apply DES three times with different keys → 168-bit effective key; legacy banking systems
  • AES (Advanced Encryption Standard): current standard (NIST 2001); 128-bit block, 10/12/14 rounds; considered secure
  • CBC (Cipher Block Chaining) mode: each plaintext block is XORed with previous ciphertext block before encryption. Uses IV.
  • CTR (Counter) mode: turns block cipher into stream cipher by encrypting successive counter values; parallelizable
  • Key distribution problem: to share a key securely, you need a secure channel — but you're trying to create one
INTERACTIVE SIMULATION
THINK OF IT LIKE…

Symmetric key is like a padlock where Alice and Bob both have a copy of the same key. It's fast and secure — but how did they both get a copy of the key without anyone else seeing? They'd have to meet in person first. Public key cryptography (next section) solves this by using a combination lock that anyone can lock but only Bob can open.

Public Key Cryptography & Digital Signatures

Key pair · RSA · Diffie-Hellman · Digital signature · MAC · Certificate Authority · X.509

Complete Section 2 to unlock

EXPLAIN

Public key cryptography uses a mathematically linked key pair: a public key K+ (shared openly) and a private key K− (kept secret). This enables encryption for confidentiality (encrypt with recipient's public key; only they can decrypt) and digital signatures for authentication and integrity (sign with your private key; anyone can verify with your public key).

  • Key pair: K+ (public, shareable) and K− (private, secret) — mathematically related
  • Encryption for confidentiality: E(K+_Bob, m) — only Bob can decrypt with K−_Bob
  • Digital signature: S = E(K−_Alice, m) — anyone can verify with K+_Alice; proves Alice created it
  • RSA: security based on difficulty of factoring large integers (n = p×q). Key size: 2048 or 4096 bits
  • Diffie-Hellman key exchange: allows two parties to establish a shared secret over a public channel
  • Digital signature scheme: sign hash H(m) rather than full message (RSA key size limits direct signing)
  • MAC (Message Authentication Code): symmetric equivalent — MAC = H(K_s || m). Faster but requires shared secret
  • Certificate Authority (CA): trusted third party that signs X.509 certificates binding public keys to identities
  • Certificate chain: root CAs are pre-installed in browsers; root CA signs intermediate CAs which sign end-entity certs
INTERACTIVE SIMULATION
THINK OF IT LIKE…

A public/private key pair is like a padlock (public key) and its unique key (private key). Anyone can lock a message in your padlock (encrypt with your public key), but only you have the key to open it. A digital signature is the reverse: you lock something with YOUR private key. Anyone with your padlock can open it and confirm it must have been you who locked it.

TLS/SSL & HTTPS

TLS 1.3 handshake · Forward secrecy · ECDHE · AEAD · Certificate verification · HSTS · 0-RTT

Complete Section 3 to unlock

EXPLAIN

TLS (Transport Layer Security) secures HTTPS and most modern Internet communications. It provides confidentiality (encryption), server authentication (X.509 certificates), and message integrity (MAC). TLS sits between TCP and the application layer. TLS 1.3 (2018) is the current standard, simplifying the handshake to 1-RTT and mandating forward secrecy.

  • TLS provides: confidentiality (AES-GCM encryption), authentication (X.509 cert + CA), integrity (AEAD)
  • TLS operates between TCP and the application layer — adds security on top of reliable byte-stream
  • TLS 1.3 handshake: ClientHello (key_share) → ServerHello + Certificate + Finished → Client Finished — 1 RTT!
  • TLS 1.3 vs 1.2: 1.3 is faster (1-RTT vs 2-RTT), mandates forward secrecy (ECDHE), removes weak ciphers
  • Forward secrecy: compromise of long-term private key does NOT expose past recorded sessions (ephemeral ECDHE keys)
  • Session resumption: TLS 1.3 supports 0-RTT for repeat connections (encrypted data with first packet)
  • Certificate verification: check cert chain to trusted root CA, expiry date, hostname match, revocation (OCSP/CRL)
  • HTTPS = HTTP over TLS. Uses port 443 by default.
  • HSTS (HTTP Strict Transport Security): tells browser to always use HTTPS; prevents downgrade attacks
INTERACTIVE SIMULATION
THINK OF IT LIKE…

TLS is like sending a letter in a bank vault that gets assembled around the letter in front of you. The bank (CA) verified who the vault belongs to. The combination is mathematically derived during a brief conversation with the vault owner (handshake). Even if someone recorded your whole conversation, they can't figure out the combination — that's forward secrecy.

Firewalls & Intrusion Detection Systems (IDS)

Packet filtering · Stateful firewall · Application gateway · ACL · DMZ · IDS · IPS · Signature vs anomaly detection

Complete Section 4 to unlock

EXPLAIN

Firewalls and IDS are perimeter defenses that protect networks from external threats. A firewall controls traffic based on configured rules (ACLs). A stateful firewall tracks connection state so it can allow reply traffic without permitting arbitrary inbound connections. An IDS monitors traffic patterns to detect attacks that firewalls miss.

  • Firewall goal: allow legitimate traffic, block malicious or unauthorized traffic
  • Packet filtering (stateless): examine each packet independently — filter on: src/dest IP, port, protocol, TCP flags
  • Stateful packet filtering: track TCP connection state; only allow inbound packets that belong to established outbound connections
  • Application gateway (proxy firewall): understand application-layer protocols (HTTP, FTP, DNS); can inspect content
  • ACL (Access Control List): ordered list of rules; first matching rule wins. Implicit deny-all at the end
  • DMZ (Demilitarized Zone): subnet between Internet and internal network where public servers are placed
  • IDS (Intrusion Detection System): passive monitoring — alerts on suspicious traffic but does NOT block
  • IPS (Intrusion Prevention System): active — can block or rate-limit suspicious traffic inline
  • Signature-based detection: match traffic against known attack patterns (e.g. Snort rules). Fast but misses zero-days
  • Anomaly-based detection: learn baseline of normal traffic, alert on deviations. Can detect new attacks but higher false positive rate
INTERACTIVE SIMULATION
THINK OF IT LIKE…

A stateless firewall is like a bouncer who checks everyone's ID but doesn't remember who already went in. A stateful firewall is a smarter bouncer who keeps a guest list — if you called ahead (outbound connection), you can come in; random visitors without an invitation are turned away. An IDS is a security camera that records suspicious behavior and alerts staff. An IPS is a camera with an automatic door lock.
