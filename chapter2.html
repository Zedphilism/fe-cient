Client-Server Architecture

Always-on server · Fixed IP · Data centers · No direct client-to-client

EXPLAIN

In client-server architecture there is an always-on server with a permanent, well-known IP address that services requests from many clients. Clients do not communicate directly with each other — all traffic flows through the server.

  • Server: always-on host with a fixed, well-known IP; passively waits for and responds to requests
  • Client: initiates contact; can have dynamic IP addresses
  • Clients do NOT communicate directly with each other in pure client-server
  • A single server cannot handle millions of requests — data centers aggregate thousands of servers
  • Examples: web (HTTP), email (SMTP), file transfer (FTP), remote login (SSH)
  • Scales by adding servers, but infrastructure cost is centralised and high
INTERACTIVE SIMULATION
THINK OF IT LIKE…

A restaurant. The kitchen (server) is always open and accepts orders from diners (clients). Diners never cook for each other — all service flows through the kitchen. If demand grows, the restaurant opens more kitchens (data centers).

Peer-to-Peer (P2P) Architecture

Self-scalability · Peers as client + server · BitTorrent · Challenges

Complete Section 1 to unlock

EXPLAIN

In P2P architecture, peers communicate directly with each other without a dedicated server. Each peer is simultaneously a client and a server. Adding more peers increases both demand and capacity — this is called self-scalability.

  • No always-on dedicated server — peers communicate directly
  • Each peer is both client (requests) and server (provides resources to others)
  • Self-scalability: every new peer adds capacity in proportion to its demand
  • Examples: BitTorrent, Gnutella, original Skype, blockchain networks
  • Challenges: dynamic IP addresses, intermittent peer availability, security, complex peer discovery
INTERACTIVE SIMULATION
THINK OF IT LIKE…

A neighbourhood book exchange. Instead of one central library (server), every household has books and anyone can borrow from any neighbour. The more households join, the more books are available — the system gets better as it grows, not worse.

Processes and Sockets

Process vs host · Socket API · Application-transport interface · IPC

Complete Section 2 to unlock

EXPLAIN

Network communication happens between processes (running programs), not between computers. Processes on different hosts exchange messages through sockets — the software interface (API) between the application layer and the transport layer.

  • A process is a program running within an end system
  • Within a host: processes communicate via OS inter-process communication (IPC)
  • Across hosts: processes exchange messages through sockets
  • A socket is the door through which a process sends and receives network messages
  • Developer controls the application side of the socket; OS controls transport and below
  • In client-server: the process that initiates contact = client; the one that waits = server
INTERACTIVE SIMULATION
THINK OF IT LIKE…

A socket is like a mailbox slot in your front door. Your application (process) writes letters (messages) and drops them in the slot (socket). The postal system (transport/network layer) handles delivery. Arriving letters fall through the recipient's slot into their home (process).

Addressing Processes

IP address · Port number · Well-known ports · 4-tuple

Complete Section 3 to unlock

EXPLAIN

To send a message to a process, you need two pieces of information: the IP address of the destination host and a port number identifying the specific process. Together they uniquely identify one endpoint on the Internet.

  • IP address: 32-bit identifier (IPv4) identifying the host
  • Port number: 16-bit identifier (0–65535) identifying the process on that host
  • Well-known ports: HTTP = 80, HTTPS = 443, SMTP = 25, FTP = 21, SSH = 22, DNS = 53
  • A TCP connection is uniquely identified by a 4-tuple: (src IP, src port, dst IP, dst port)
  • Client ports are assigned ephemerally (randomly) by the OS; server ports are fixed/well-known
  • Ports 0–1023: reserved for well-known services; 49152–65535: dynamic/private
INTERACTIVE SIMULATION
THINK OF IT LIKE…

An IP address is the street address of an apartment building — it gets you to the right building. The port number is the apartment number — it identifies the exact unit (process) within that building. You need both to deliver a message to the right person.

Application-Layer Protocols

Message types · Syntax · Semantics · Open vs proprietary · RFC

Complete Section 4 to unlock

EXPLAIN

An application-layer protocol defines how processes on different hosts pass messages to each other: message types, syntax (fields and structure), semantics (meaning of fields), and rules for when/how to send and respond.

  • Defines: message types (request/response), syntax, semantics, and communication rules
  • Open protocols (RFC-defined): HTTP, SMTP, FTP, DNS — enable interoperability
  • Proprietary protocols: specific to one vendor (original Skype) — lock users in
  • The application is distinct from its protocol: email ≠ SMTP; the web ≠ HTTP
  • Application-layer protocols rely on transport services provided by TCP or UDP below
INTERACTIVE SIMULATION
THINK OF IT LIKE…

A protocol is like the rules of a formal business letter: standard sections (To, From, Subject, Body), agreed language conventions, rules about when to reply. If everyone follows the same format, any company can correspond with any other — that's interoperability through open protocols.

Transport Service Requirements

Data integrity · Throughput · Timing · Security · Choosing TCP vs UDP

Complete Section 5 to unlock

EXPLAIN

Different applications have different transport needs. The four key dimensions are: data integrity (no loss), throughput (minimum bandwidth), timing (low latency), and security (encryption/auth).

  • Data integrity: file transfer, email — require 100% reliable delivery; audio/video can tolerate loss
  • Throughput: video conferencing needs minimum guaranteed rate; email is elastic (uses whatever is available)
  • Timing: VoIP, gaming require <100–150 ms delay; non-real-time apps tolerate delay
  • Security: e-commerce needs encryption, data integrity, endpoint authentication
  • TCP: reliable delivery, flow/congestion control — no timing or throughput guarantees
  • UDP: unreliable, connectionless, minimal overhead — used when timing > reliability
INTERACTIVE SIMULATION
THINK OF IT LIKE…

Choosing a transport service is like choosing how to ship a package. For rare documents (financial data), use certified mail (TCP) — expensive but reliable. For a live phone call (VoIP), you want immediacy, not a retry — use a phone call (UDP). If a word is cut off, you'd rather continue than wait.

TCP vs UDP

Connection-oriented · Reliable delivery · Flow control · Connectionless · Fire-and-forget

Complete Section 6 to unlock

EXPLAIN

TCP provides reliable, connection-oriented delivery with flow and congestion control. UDP provides unreliable, connectionless delivery with minimal overhead. Neither guarantees bandwidth or timing.

  • TCP: connection-oriented — 3-way handshake before data transfer
  • TCP: reliable, ordered, error-checked delivery; retransmits lost segments
  • TCP: flow control — won't overwhelm receiver's buffer (receive window)
  • TCP: congestion control — throttles sender when network is congested
  • UDP: connectionless — no handshake; sends datagrams immediately
  • UDP: unreliable — no guarantee of delivery, order, or integrity
  • UDP: no flow/congestion control — sends at whatever rate the application sets
  • UDP advantages: lower overhead, lower latency, supports broadcast/multicast

THINK OF IT LIKE…

TCP is a registered letter with tracking and signature confirmation — reliable but with overhead. UDP is dropping a postcard in a public mailbox — fast and cheap, but no guarantee it arrives. Use registered mail for your tax return; use postcards when speed matters more than certainty.

Web and HTTP — Basics

URL structure · Web objects · TCP port 80 · Stateless protocol

Complete Section 7 to unlock

EXPLAIN

The World Wide Web uses HTTP (HyperText Transfer Protocol) as its application-layer protocol. A web page is made up of objects — each a file addressable by a URL. HTTP defines how browsers request objects and how servers respond, running over TCP on port 80.

  • HTTP defined in RFC 1945 (v1.0) and RFC 2616 (v1.1)
  • A web page = 1 base HTML file + referenced objects (images, CSS, JS)
  • URL structure: protocol://hostname/path — e.g., http://www.example.com/img/logo.png
  • HTTP runs over TCP on port 80; HTTPS on port 443
  • HTTP is stateless — the server retains no memory of past client requests
  • Statelessness simplifies server design; cookies layer state on top when needed
INTERACTIVE SIMULATION
THINK OF IT LIKE…

HTTP is like ordering at a fast-food counter. You (client) say "GET me one burger" (HTTP request). The server hands you the burger (HTTP response). The cashier remembers nothing about you after you walk away (stateless). A loyalty card (cookies) is a separate system layered on top.

HTTP Characteristics

Methods (GET/POST) · Status codes · Headers · Cookies · Web cache

Complete Section 8 to unlock

EXPLAIN

HTTP uses ASCII text messages with a clear structure. Requests have a method, URL, headers, and optional body. Responses have a status code, headers, and body. Cookies add state; web caches reduce latency and origin server load.

  • Request methods: GET (retrieve), POST (submit data), PUT (upload), DELETE, HEAD
  • Status codes: 200 OK, 301 Moved Permanently, 400 Bad Request, 404 Not Found, 500 Internal Server Error
  • Key request headers: Host, Connection, User-Agent, Accept-Language
  • Key response headers: Content-Type, Content-Length, Date, Last-Modified
  • Cookies: 4-component mechanism (Set-Cookie header, cookie file, server DB, Cookie header) — layers state on stateless HTTP
  • Web cache (proxy): stores recently requested objects locally; serves them without contacting the origin server
INTERACTIVE SIMULATION
THINK OF IT LIKE…

HTTP status codes are like restaurant responses. 200 OK = "Here's your order". 301 Moved = "We've moved, here's the new address". 404 Not Found = "We don't have that dish". 500 Server Error = "Our kitchen just caught fire — sorry".

Persistent vs Non-Persistent HTTP

RTT cost · 2 RTTs per object · Pipelining · HTTP/1.1 default

Complete Section 9 to unlock

EXPLAIN

Non-persistent HTTP opens a new TCP connection for every object — costing 2 RTTs per object. Persistent HTTP (default in HTTP/1.1) keeps the connection open across requests. Pipelining further allows multiple requests to be sent without waiting for responses.

  • RTT (Round-Trip Time): time for a small packet to travel client → server → client
  • Non-persistent: 1 RTT for TCP handshake + 1 RTT for HTTP request/response = 2 RTTs per object
  • For N objects: non-persistent requires 2N RTTs total (ignoring transmission time)
  • Persistent HTTP: server keeps TCP connection open; subsequent requests use the same connection
  • Persistent + pipelining: all requests sent back-to-back; as little as 1 RTT for all objects
  • HTTP/2: true multiplexing — multiple parallel streams on one connection + header compression
INTERACTIVE SIMULATION
THINK OF IT LIKE…

Non-persistent = separate phone call to order each item — hang up, call again. Persistent without pipelining = one call, order one item, wait for confirmation, then order next. Persistent + pipelining = one call, rattle off your entire order at once. Much faster.

HTTP Request-Response Flow

6-step transaction · Request line · Status line · Content-Type · Content-Length

Complete Section 10 to unlock

EXPLAIN

An HTTP transaction follows a clear request-response pattern over TCP. The client sends a request with method, URL, headers, and optional body. The server responds with a status code, headers, and the requested object.

  • Step 1: Client opens TCP connection to server port 80
  • Step 2: Server accepts the TCP connection
  • Step 3: Client sends HTTP request (e.g., GET /index.html HTTP/1.1)
  • Step 4: Server retrieves the object and wraps it in an HTTP response
  • Step 5: Server sends HTTP response (e.g., HTTP/1.1 200 OK + body)
  • Step 6: Connection closed (non-persistent) or kept open (persistent)
  • Request line: method SP URL SP version CRLF
  • Status line: version SP status-code SP phrase CRLF
  • Content-Type tells the browser how to render the response body
INTERACTIVE SIMULATION
THINK OF IT LIKE…

HTTP flow is like a drive-through order. Pull up (TCP connect). Speak: "GET large fries" (HTTP GET request). Server repeats your order back and hands it through the window with a receipt — "200 OK — here are your fries and the nutritional info" (response with headers + body). Then drive off.

Electronic Mail

SMTP · IMAP · POP3 · Mail servers · User agents · Store-and-forward

Complete Section 11 to unlock

EXPLAIN

Email has three major components: user agents (mail clients), mail servers (hold mailboxes and outgoing queues), and protocols. SMTP pushes mail between servers. IMAP and POP3 let users pull mail from their mail server.

  • User agent: mail client (Outlook, Thunderbird, Gmail web UI) for composing, reading, sending
  • Mail server: holds the user's mailbox (incoming) and a message queue (outgoing)
  • SMTP: push protocol; transfers mail from sending to receiving mail server over TCP port 25
  • SMTP messages are 7-bit ASCII; binary attachments use MIME (Base64 encoding)
  • POP3: downloads messages to client, typically removes from server — single device
  • IMAP: keeps messages on server, syncs state across all devices — preferred for multi-device use
  • Send path: user agent → Alice's server (SMTP) → Bob's server (SMTP) → Bob retrieves (IMAP/POP3)
INTERACTIVE SIMULATION
THINK OF IT LIKE…

Email uses two services. SMTP is the postal delivery driver — picks up your outgoing letter and delivers it to the recipient's post office box (mail server). IMAP leaves copies at the post office for multi-device access. POP3 lets you take everything home to one device.
